## 2. 주요 발견 사항

### 2.1 색상 팔레트 과다 (심각도: 높음)

427개의 고유 색상은 디자인 시스템 기준으로 매우 과다하다. 동일한 "회색"이 #64748B, #475569, #6B7280, #94A3B8 등 미묘하게 다른 값으로 반복 사용되고 있어 시각적 통일감이 떨어진다.

**현재 문제점:**
- 비슷한 회색 계열이 4~5종씩 혼용
- 탭별 컬러(보라, 초록, 파랑, 주황)는 잘 구분되나, 보조 색상이 일관성 없이 사용
- 상태 색상(성공/경고/에러)이 컴포넌트마다 미묘하게 다른 값 사용

**개선 방향:**
- 핵심 팔레트를 20~30개로 정의하고 CSS 변수화
- 탭별 primary/secondary/accent 3단계로 정리
- 상태 색상(success, warning, error, info) 통일

### 2.2 border-radius 불일치 (심각도: 중간)

6px, 8px, 10px, 12px, 14px, 16px, 18px, 20px, 24px, 28px, 999px 등 15종 이상의 border-radius가 혼재한다. 같은 카드 컴포넌트인데 어떤 곳은 12px, 다른 곳은 14px, 또 다른 곳은 16px을 사용하는 식이다.

**개선 방향:**
- 3~4단계로 통일: `--radius-sm: 8px`, `--radius-md: 12px`, `--radius-lg: 20px`, `--radius-full: 999px`
- 용도별 가이드: 버튼(sm), 카드(md), 모달(lg), 뱃지/아바타(full)

### 2.3 font-size 과다 (심각도: 중간)

9px부터 48px까지 16단계의 font-size가 사용되고 있다. 디자인 시스템에서는 보통 5~7단계(xs, sm, base, md, lg, xl, 2xl)면 충분하다.

**개선 방향:**
- 타입 스케일 정의: `--text-xs: 11px`, `--text-sm: 12px`, `--text-base: 14px`, `--text-md: 16px`, `--text-lg: 20px`, `--text-xl: 24px`
- 13px(가장 많이 사용)을 기본 크기로 설정할지, 14px로 통일할지 결정 필요

### 2.4 히어로 영역 정보 밀도 (심각도: 낮음)

홈 탭의 상단 히어로 영역에 인사말, 연속일수, 총 마일리지, 등급 정보가 모두 표시된다. 정보가 풍부한 것은 좋으나, 시각적 무게가 과해 "어디를 먼저 봐야 하지?"라는 느낌을 줄 수 있다.

**개선 방향:**
- 핵심 정보(연속일수 또는 총 마일리지)에 시각적 위계를 더 부여
- 보조 정보는 크기/투명도로 위계 차이를 줄 것

### 2.5 그라디언트 과다 사용 (심각도: 낮음)

92회의 그라디언트 사용은 앱이 화려해 보이지만, 모든 요소가 그라디언트를 가지면 오히려 주목성이 분산된다.

**개선 방향:**
- 히어로/CTA 등 핵심 영역에만 그라디언트 사용
- 일반 카드/리스트 아이템은 단색 배경으로 단순화
- 그라디언트 사용처를 현재의 1/3 수준으로 축소

### 2.6 컴포넌트 스타일 불일치 (심각도: 중간)

동일한 역할의 컴포넌트(카드, 버튼, 리스트 아이템)가 탭마다 다른 스타일을 가진다. 예를 들어 카드의 그림자, 패딩, 테두리 색상이 화면마다 조금씩 다르다.

**개선 방향:**
- 공통 컴포넌트 클래스 정의: `.card`, `.card-sm`, `.btn`, `.btn-sm`, `.list-item`
- 탭별 차이는 `--tab-color` 변수만으로 처리하고 구조는 동일하게

## 3. 긍정적 요소

현재 앱에서 디자인적으로 잘 되어 있는 부분들도 있다:

- **탭별 컬러 시스템**: `body[data-tab]`으로 탭별 테마 색상을 전환하는 구조는 잘 설계됨
- **다크모드 지원**: CSS 변수 기반 다크모드가 잘 작동
- **하단 네비게이션 바**: 깔끔하고 직관적인 아이콘 + 라벨 조합
- **뱃지/곤충 SVG**: 개성 있는 캐릭터 디자인으로 앱의 재미 요소 제공
- **게이미피케이션 요소**: 연속일수, 등급, 마일리지 등 동기부여 요소가 잘 배치됨

## 4. 단계별 개선 로드맵

### Phase 1: 색상 팔레트 정리 — ✅ 완료 (2026-04-14, v0414a, PR #335/#336)

**작업 결과:**
1. ✅ 누락 색상 변수 12개 추가 (--or-dk, --or-bright, --gn-bright, --rd-dk, --pu-bright, --pu-dk, --yw-dk, --yw-bright 등)
2. ✅ 하드코딩 색상 349개를 CSS 변수(var())로 교체
3. ✅ CSS 변수 사용률 742→1,752회 (+136%), 하드코딩 색상 693→344개 (-50%)
4. ✅ 다크모드 전용 값은 유지, 테마 반응형 색상만 교체

**남은 작업:** 다크모드 규칙 내 하드코딩 값 344개 (순환참조 방지를 위해 유지)

### Phase 2: 디자인 토큰 표준화 — ✅ 완료 (2026-04-14, v0414a, PR #335/#336)

**작업 결과:**
1. ✅ border-radius 15종 → 7단계 통합 (--rxs 8px ~ --rfull 999px), 185개 교체
2. ✅ font-size 16단계 → 9단계 통합 (--fs-xs 10px ~ --fs-4xl 28px), 410개 교체
3. ⏳ spacing(margin/padding) 기준값 정의 — Phase 2.5로 이월
4. ✅ box-shadow 3단계 체계 (--shadow-sm/--shadow/--shadow-lg) 변수 정의

**남은 작업:** spacing 토큰화, box-shadow 자동 교체 (맥락 의존성으로 수동 교체 권장)

### Phase 3: 컴포넌트 스타일 통일 (안전도: 중간, 영향: 큼)

**작업 내용:**
1. 공통 CSS 클래스 정의 (`.card`, `.btn`, `.list-item` 등)
2. 인라인 스타일을 클래스 기반으로 전환
3. 탭별 변형은 CSS 변수로만 처리

**예상 소요:** 3~5세션  
**위험도:** 중간 (HTML 구조 변경 포함, 충분한 테스트 필요)

### Phase 4: 정보 위계 및 레이아웃 개선 (안전도: 낮음, 영향: 큼)

**작업 내용:**
1. 히어로 영역 정보 위계 재설계
2. 그라디언트 사용 축소 및 정리
3. 카드 간 여백/간격 통일

**예상 소요:** 2~3세션  
**위험도:** 중간~높음 (레이아웃 변경은 예기치 않은 깨짐 가능)

## 5. 개선 원칙

1. **기능 먼저, 디자인 다음**: 기존 기능이 깨지지 않는 범위 내에서만 진행
2. **점진적 변경**: 한 번에 하나의 영역만 수정하고 검증
3. **feature 브랜치 사용**: 모든 CSS 변경도 반드시 feature 브랜치 → PR → merge
4. **시각 검증**: 변경 전후 스크린샷 비교로 의도치 않은 깨짐 확인
5. **다크모드 동시 검증**: 라이트/다크 모드 양쪽 모두 확인
